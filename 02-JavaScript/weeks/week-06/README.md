# 6주차: TypeScript 정적 바인딩 및 클린 코드 도입

## 📖 핵심 요약
자바스크립트의 유연함이 대규모 서비스 개발 시 버그의 온상이 될 수 있음을 인지하고, 안정적인 타입 시스템인 **TypeScript**로의 전환(마이그레이션)을 직접 실습합니다. 또한 클린 코드 및 유닛 테스트(Unit Test)를 통해 코드 검증 안정성을 높이는 방법을 학습합니다.

*   **TypeScript 정적 검사**: 컴파일 시점에 타입을 대조하여 런타임 이전에 타입 오류와 예외를 잡아낼 수 있는 기술입니다.
*   **단일 책임 원칙 (SRP - Single Responsibility Principle)**: 한 클래스나 함수는 오직 하나의 변경 이유(즉, 하나의 책무)만 가져야 한다는 클린 코드 설계 법칙입니다.
*   **정적 정렬 및 검증**: Prettier와 ESLint를 유기적으로 구성하여 협업 시 동일한 코딩 스타일을 규칙으로 준수하도록 강제합니다.

## 🎯 학습 목표
1.  기존 자바스크립트 모듈을 타입 선언과 인터페이스를 반영한 TypeScript 파일(`.ts`)로 리팩토링할 수 있다.
2.  단일 책임 원칙을 적용하여 함수가 복잡한 동작을 분할하도록 수정할 수 있다.
3.  Jest 프레임워크를 기반으로 타입 안정성 및 로직의 예외 상황들을 검증하는 테스트 코드를 작성할 수 있다.

## 🧪 6주차 미션: TypeScript 기반의 API 결과 포맷터 모듈 및 단위 테스트 구현
*   **미션 파일**: `mission.ts`
*   **요구 사항**:
    1.  API로부터 내려받는 날것의 데이터 규격인 `RawApiResponse` 인터페이스를 선언합니다.
    2.  정제 완료되어 프런트엔드에서 활용될 규격인 `CleanedData` 인터페이스를 선언합니다.
    3.  `formatApiResponse(rawResponse: RawApiResponse): CleanedData` 함수를 설계하되, 날짜 포맷 변환, 널값(Nullish) 디폴트 처리, 필수 필드 누락 시의 커스텀 예외 처리를 정적 타입 보장 하에 구현하십시오.
    4.  동일 디렉터리 내의 `mission.test.ts`를 작성하여 테스트 커버리지를 채우고 컴파일 에러가 없는 환경을 구성합니다.
